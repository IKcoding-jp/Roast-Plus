# スキル監査スキル

スキルファイルは放置すると陳腐化・重複が発生する。このスキルを定期実行することで、スキルコレクションを最適な状態に保つ。

## 実行手順

### Step 1: スキル一覧の取得

```bash
ls .claude/skills/
```

各スキルの `SKILL.md`（および `references/` 内のファイル）を読み込む。

---

### Step 2: 各スキルを6観点で評価

| 観点 | チェック内容 |
|------|-------------|
| **コードベース整合性** | スキルが参照するファイル・パターン・コマンドが現在のコードベースに存在するか |
| **CLAUDE.mdとの重複** | CLAUDE.mdのルール・セクションで代替できる内容を含んでいないか |
| **スキル間の重複** | 他のスキルと機能が重複していないか（対象スキルを特定） |
| **内容の鮮度** | 現在の開発フロー・ライブラリバージョン・ファイル構造と一致しているか |
| **使用頻度** | 実際にどれくらい呼び出されているか（下記の測定方法で調査） |
| **投資対効果（ROI）** | コンテキストウィンドウのコストに見合う価値を提供しているか |

#### 使用頻度の測定方法

以下の手順で各スキルの実使用状況を調査する:

1. **セッションログ検索**（最も正確）:
   ```bash
   grep -r "skill_name" ~/.claude/projects/*/transcripts/ --include="*.jsonl" -l | wc -l
   ```
2. **CLAUDE.md参照**: スキルがCLAUDE.mdのワークフローに組み込まれているか（組み込み済み＝高頻度の指標）
3. **git履歴**: スキル自体の更新頻度（`git log --oneline -- .claude/skills/<skill>/`）

頻度は3段階で分類:
- **高**: 週1回以上、またはCLAUDE.mdのワークフローに組み込み済み
- **中**: 月1〜3回
- **低**: 月1回未満、または使用実績なし

#### 投資対効果（ROI）の評価基準

| 基準 | 高ROI | 低ROI |
|------|-------|-------|
| **代替不可能性** | Claudeの基本能力では補えない固有知識 | 一般的なプログラミング知識で代替可能 |
| **エラー防止** | スキルなしでは頻発するミスを防止 | あっても結果が変わらない |
| **プロジェクト固有性** | このプロジェクト特有のパターン・規約 | 汎用的な内容 |
| **コンテキストコスト** | SKILL.md + references が簡潔（〜200行） | 大量のトークンを消費（500行超） |

---

### Step 3: 判定

各スキルを以下のいずれかに分類する:

| 判定 | 基準 |
|------|------|
| **✅ KEEP** | 現在のコードベースと整合し、独自の価値がある |
| **⚠️ UPDATE** | 内容は有効だが、更新が必要な箇所がある（具体箇所を明記） |
| **🔀 MERGE** | 別のスキルと統合可能（統合先スキルを明記） |
| **❌ DELETE** | コードベースと乖離、またはCLAUDE.mdで完全に代替可能 |

#### 頻度×ROI マトリクス（判定の補助指標）

従来の4観点（整合性・重複・鮮度）に加え、頻度とROIの組み合わせで判定を補強する:

|  | **高ROI** | **低ROI** |
|---|-----------|-----------|
| **高頻度** | ✅ KEEP（最優先で維持） | 🔀 MERGE（CLAUDE.mdへの統合を検討） |
| **中頻度** | ✅ KEEP | ⚠️ UPDATE（スリム化 or トリガー条件改善） |
| **低頻度** | ⚠️ UPDATE（トリガー条件を見直し） | ❌ DELETE候補 |

⚠️ マトリクスは補助指標。最終判定は6観点すべてを総合して決定する。
例: 低頻度・高ROIのスキルは、使われるべき場面で発火していない可能性がある → description改善を検討

---

### Step 4: 結果サマリーを提示

以下の形式でユーザーに提示する:

```
📋 スキル監査結果 (YYYY-MM-DD)
========================================

| スキル           | 頻度 | ROI | 判定        | 理由                               |
|-----------------|:----:|:---:|-------------|-----------------------------------|
| fix-issue       | 高   | 高  | ✅ KEEP     | 現行フローと整合、独自価値あり       |
| roastplus-ui    | 中   | 高  | ✅ KEEP     | デザインシステム参照として必須       |
| example-skill   | 低   | 低  | ❌ DELETE   | CLAUDE.mdのUI Rulesで完全代替可能  |
| old-skill       | 中   | 中  | ⚠️ UPDATE   | Next.js 16対応への更新が必要       |
| rarely-used     | 低   | 高  | ⚠️ UPDATE   | description改善でトリガー率向上を  |

## 推奨アクション
1. `example-skill` を削除（低頻度・低ROI）
2. `old-skill` のNext.js 16対応箇所を更新
3. `rarely-used` のdescriptionを改善（高ROIだがトリガーされていない）

## 要更新スキル詳細
- `old-skill`: L.24の `pages/` パスを `app/` に変更
- `rarely-used`: descriptionに具体的なトリガー文言を追加
```

---

### Step 5: ユーザー承認後に実行

ユーザーの承認を得てから削除・更新・マージを実行する。

⚠️ **削除・更新の前に必ず承認を取る。自動実行禁止。**

承認後、判定別に以下の手順で実行する:

| 判定 | 実行方法 |
|------|---------|
| **❌ DELETE** | スキルディレクトリをそのまま削除 |
| **⚠️ UPDATE** | `/skill-creator` スキルを呼び出し、Step 4（Edit the Skill）から実行 |
| **🔀 MERGE** | `/skill-creator` スキルを呼び出し、統合先スキルのStep 4（Edit the Skill）から実行。元スキルを削除 |

**UPDATE・MERGEの実行パターン:**

```
Using skill-creator to update `<スキル名>` — Step 4: Edit the Skill から実行します
```

- Step 1〜3（理解・計画・初期化）はスキップ（スキルは既存のため）
- Step 4: 監査で特定した問題箇所を編集
- Step 5: `scripts/package_skill.py` でパッケージング（該当する場合）
- Step 6: 実行後にユーザーへ変更サマリーを提示

---

## 監査推奨タイミング

- **月1回**: 定期実行
- **スキル追加時**: 既存スキルとの重複チェック
- **大規模リファクタリング後**: コードベース整合性の再確認
- **依存ライブラリのメジャーバージョンアップ時**: APIの破壊的変更チェック
