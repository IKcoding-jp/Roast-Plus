## requirement.md（要件定義）

```markdown
# 要件定義

**Issue**: #[番号]
**作成日**: YYYY-MM-DD
**ラベル**: [ラベル]

## ユーザーストーリー

ユーザー「[セリフ形式で、何をしたいか]」
アプリ「[期待する動作]」

## 要件一覧

### 必須要件
- [ ] 要件1 - 具体的な動作
- [ ] 要件2 - 具体的な動作

### オプション要件
- [ ] 要件3 - あれば望ましい動作

## 非機能要件
- パフォーマンス: [目標値]
- アクセシビリティ: [対応内容]
- モバイル対応: [レスポンシブ要件]

## 受け入れ基準
- [ ] 基準1
- [ ] 基準2

## 参照
- 関連Issue: #xxx
- Steering Documents: FEATURES.md「[機能名]」
```

### 生成ガイド

- Issue本文の「要件」「背景」セクションをパース
- FEATURES.mdから関連機能の仕様を参照して整合性確認
- 受け入れ基準はIssue本文のチェックリストから抽出、不足分は補完

## tasklist.md（タスクリスト）

```markdown
# タスクリスト

## フェーズ1: [フェーズ名]（例: データモデル実装）
- [ ] タスク1
  - [ ] サブタスク1-1
  - [ ] サブタスク1-2
- [ ] タスク2

## フェーズ2: [フェーズ名]（例: UI実装）
- [ ] タスク3
- [ ] タスク4

## フェーズ3: [フェーズ名]（例: テスト）
- [ ] ユニットテスト
- [ ] 統合テスト

## 依存関係
- フェーズ1 → フェーズ2 → フェーズ3（順次実行）

## 見積もり
- フェーズ1: [時間]
- フェーズ2: [時間]
- **合計**: [時間]
```

### 生成ガイド

- design.mdの「変更対象ファイル」「新規作成ファイル」からタスクを導出
- GUIDELINES.mdの実装フロー（型定義 → ロジック → UI → テスト）に従う順序
- 依存関係を明示（フェーズ間のブロック）
- 見積もりはAIエージェント実行前提（人間の作業時間ではない）

## design.md（設計書）

```markdown
# 設計書

## 実装方針

### 変更対象ファイル
- `path/to/file.ts` - [変更内容]

### 新規作成ファイル
- `path/to/new.ts` - [役割]

## データモデル

### Firestoreスキーマ（該当する場合）
[TypeScript interface定義]

## UI設計（該当する場合）

### コンポーネント構成
- `ComponentName` (既存) → [変更内容]
- `NewComponent` (新規) → [役割]

### 使用する共通コンポーネント
- `@/components/ui` から使用するコンポーネントを列挙

## API設計（該当する場合）

### 関数シグネチャ
[TypeScript関数定義]

## 影響範囲
- [影響を受けるコンポーネント・機能]

## UI実装ルール確認
- [ ] 共通コンポーネント使用（`@/components/ui`）
- [ ] テーマ対応: セマンティックCSS変数使用（`bg-page`, `text-ink` 等）
- [ ] ハードコード色の禁止（`bg-white`, `text-gray-800` 等は非推奨）

## 禁止事項チェック
- [ ] 独自CSSでボタン/カード/入力を作らない
- [ ] 新しい状態管理ライブラリを導入しない
- [ ] 設計方針を変更しない（変更が必要な場合は相談）

## ADR（この設計の決定事項）

### Decision-001: [タイトル]
- **理由**: [理由]
- **影響**: [影響]
```

### 生成ガイド

- Serena MCP調査結果から変更対象ファイルを特定
- GUIDELINES.md, TECH_SPEC.mdから実装パターンを参照
- FEATURES.mdの「禁止事項」を反映
- テーマ対応は `data-theme` 属性 + CSS変数方式（テーマ関連のpropは不要）
- 共通コンポーネントの確認: `components/ui/index.ts` のエクスポート一覧を参照
- 不要なセクションは省略可（例: Firestore不使用ならデータモデル省略）

## testing.md（テスト計画）

```markdown
# テスト計画

## テスト戦略

### ユニットテスト（Vitest）
- `path/to/file.test.ts`
  - テストケース1
  - テストケース2（境界値）
  - テストケース3（異常系）

### 統合テスト（Testing Library）
- `path/to/component.test.tsx`
  - UI操作シナリオ
  - 状態遷移テスト

### E2Eテスト（手動 or Playwright MCP）
- スクリーンショット確認項目

## テストケース一覧

| テストケース | 入力 | 期待出力 |
|-------------|-----|---------|
| 正常系: [ケース名] | [入力値] | [期待値] |
| 境界値: [ケース名] | [入力値] | [期待値] |
| 異常系: [ケース名] | [入力値] | [エラー内容] |

## カバレッジ目標
- lib/: 90%以上
- hooks/: 85%以上
- 全体: 75%以上
```

### 生成ガイド

- tasklist.mdのタスクからテスト対象を導出
- GUIDELINES.mdの「テスト戦略」を参照
- 境界値テスト、異常系テストを自動提案
- TDD前提: テストが実装より先に書かれることを意識したケース設計
